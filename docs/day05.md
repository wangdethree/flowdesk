# Day 5：评论记录、状态流转和分页

## 今天完成了什么

- 为工单列表开启默认分页。
- 新增工单评论/处理记录接口。
- 增加工单状态流转校验。
- 进入已解决、已关闭状态时自动记录时间。
- 调整处理人权限：处理人可以更新分配给自己的工单，但不能删除。
- 补充评论、分页、状态流转和权限测试。

## 新增接口

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET` | `/api/tickets/{id}/comments/` | 查询某张工单下的评论和处理记录 |
| `POST` | `/api/tickets/{id}/comments/` | 新增评论或处理记录 |

## 评论接口示例

请求：

```http
POST /api/tickets/1/comments/
Authorization: Bearer <access_token>
Content-Type: application/json
```

```json
{
  "content": "已经开始排查网络配置。",
  "comment_type": "handling"
}
```

说明：

- `ticket` 不需要前端传，后端会根据 URL 里的工单 ID 自动绑定。
- `author` 不需要前端传，后端会根据当前登录用户自动绑定。
- `comment_type=comment` 表示普通评论。
- `comment_type=handling` 表示处理记录。

## 分页响应结构

开启分页后，列表接口不再直接返回数组，而是返回一个对象：

```json
{
  "count": 12,
  "next": "http://127.0.0.1:8000/api/tickets/?page=2",
  "previous": null,
  "results": []
}
```

字段说明：

- `count` 表示总记录数。
- `next` 表示下一页地址，没有下一页时为 `null`。
- `previous` 表示上一页地址，没有上一页时为 `null`。
- `results` 才是真正的数据列表。

## 状态流转规则

当前允许的状态流转：

| 当前状态 | 可以流转到 |
| --- | --- |
| `open` 待处理 | `in_progress` 处理中、`closed` 已关闭 |
| `in_progress` 处理中 | `resolved` 已解决、`closed` 已关闭 |
| `resolved` 已解决 | `closed` 已关闭 |
| `closed` 已关闭 | 无 |

这样设计是为了避免工单状态被随意回滚。比如已关闭工单不能直接改回待处理，否则处理历史会变得混乱。

## 权限规则更新

- 管理员可以查看、更新、删除所有工单。
- 创建人可以查看、更新、删除自己的工单。
- 处理人可以查看和更新分配给自己的工单。
- 处理人不能删除工单。
- 无关用户看不到工单，也不能写评论。

## 核心概念

### 什么是分页？

分页是把大量列表数据拆成多页返回。这样可以减少单次响应的数据量，也能降低数据库和网络压力。

### 为什么评论接口挂在工单下面？

评论依附于某张工单，不应该脱离工单单独创建。所以接口设计为 `/api/tickets/{id}/comments/`，含义更清楚。

### 什么是状态机？

状态机是对状态和状态变化规则的约束。工单不能随便从任意状态跳到任意状态，只能按后端定义的规则流转。

### 为什么时间字段由后端维护？

`resolved_at` 和 `closed_at` 代表重要业务时间，不能让前端随便提交。后端在状态变化时自动写入，数据更可信。

## 面试题

1. 为什么列表接口需要分页？
2. DRF 分页后的响应结构是什么？
3. 为什么评论接口设计成 `/api/tickets/{id}/comments/`？
4. 什么是状态机？
5. 工单状态为什么不能随便修改？
6. 为什么 `resolved_at` 和 `closed_at` 不让前端传？
7. 创建评论时为什么不让前端传 `author`？
8. 处理人为什么可以更新但不能删除工单？

## 简短回答

1. 分页可以降低单次查询和响应压力，避免一次返回过多数据。
2. DRF 分页响应包含 `count`、`next`、`previous` 和 `results`。
3. 因为评论依附于工单，嵌套路径能表达资源关系。
4. 状态机用于约束一个对象可以有哪些状态，以及状态之间如何变化。
5. 随意修改状态会破坏业务流程和处理历史。
6. 这些关键时间应该由后端根据状态变化自动维护，避免被伪造。
7. 作者应该来自当前登录用户，不能让前端伪造。
8. 处理人负责推进处理进度，但删除工单属于更高风险操作，先限制给创建人和管理员。

## 面试时怎么讲今天的内容

我为 FlowDesk 补充了工单评论和处理记录接口，并把评论设计成工单下的嵌套资源。为了避免工单状态被随意修改，我在序列化器里增加了状态流转校验，并在进入已解决、已关闭状态时由后端自动写入时间。同时开启了 DRF 分页，保证列表接口在数据量变大后仍然稳定。权限上，普通用户只能看到相关工单，处理人可以推进分配给自己的工单，但不能删除工单。
