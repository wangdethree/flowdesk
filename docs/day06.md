## 为什么需要审计日志？

审计日志用来记录系统中的关键操作，回答这些问题：

- 谁操作了系统？
- 操作了哪类资源？
- 操作的是哪条数据？
- 做了什么动作？
- 操作发生时有哪些关键上下文？

在企业内部系统里，审计日志很常见。比如有人误删工单、错误关闭工单，或者用户反馈“不是我改的”，后端就可以通过审计日志追踪操作历史。

## 审计日志表结构

| 字段 | 作用 |
| --- | --- |
| `actor` | 操作人，可以为空，空值表示系统操作 |
| `action` | 操作类型，例如创建、更新、删除、评论、状态流转 |
| `target_type` | 目标资源类型，例如 `Ticket` |
| `target_id` | 目标资源 ID |
| `description` | 操作描述 |
| `metadata` | 扩展数据，使用 JSON 保存上下文 |
| `created_at` | 操作时间 |

## 为什么要用服务函数？

我们没有在业务代码里到处直接写：

```python
AuditLog.objects.create(...)
```

而是统一调用：

```python
create_audit_log(...)
```

这样做有两个好处：

1. 业务代码更清晰，不需要关心审计日志表的细节。
2. 后续如果要改成 Celery 异步写日志，只需要改服务函数内部实现。

## 核心概念

### 什么是审计日志？

审计日志是系统对关键操作的留痕记录。它通常不参与主业务流程展示，但对排查问题、追踪责任、风控和合规很有用。

### 什么是 metadata？

`metadata` 是 JSON 扩展字段，用来保存不同操作的额外上下文。比如创建工单时记录标题和状态，评论时记录评论 ID 和评论类型。

### 为什么删除前要先写日志？

删除后业务对象可能不存在了。如果删除之后再写日志，就拿不到标题、状态等上下文。所以删除操作要先写日志，再真正删除数据。

### 为什么审计日志的 actor 可以为空？

有些操作可能不是用户触发的，而是系统定时任务触发的。比如后续 Celery 自动关闭超时工单，这类日志就可以把 `actor` 设为空，表示系统操作。

## 面试题

1. 什么是审计日志？
2. 审计日志和普通业务日志有什么区别？
3. 为什么审计日志要记录 actor、action、target？
4. 为什么 metadata 适合用 JSONField？
5. 为什么要用服务函数封装写日志逻辑？
6. 删除数据时为什么要先写审计日志？
7. 如果日志写入影响接口性能，怎么优化？

## 简短回答

1. 审计日志是系统关键操作的留痕，用于追踪谁在什么时候做了什么。
2. 普通业务日志更多用于排查程序运行问题，审计日志更关注业务操作历史和责任追踪。
3. 这三个字段能描述一次操作的主体、动作和对象。
4. 不同操作的上下文不同，JSONField 更灵活。
5. 服务函数可以集中维护写日志逻辑，后续也方便改成异步。
6. 删除后对象上下文可能丢失，所以要先记录再删除。
7. 可以把写日志改成异步任务，例如使用 Celery 放到后台处理。

## 面试时怎么讲今天的内容

我在 FlowDesk 中新增了审计日志模块，用 `AuditLog` 记录用户对工单的关键操作，包括创建、更新、状态流转、删除和评论。业务代码不直接操作日志模型，而是通过 `create_audit_log()` 服务函数统一写入，便于后续替换成 Celery 异步写日志。删除工单时会先记录审计日志再删除，避免丢失操作上下文。
